In Java, there are two primary ways to create a thread, one is a legacy approach and the other is the industry standard.

You need to know both, but you need to know *why* we prefer one over the other.

---

## 1. Extending the `Thread` Class

This is the most straightforward way. You create a new class that inherits from the `java.lang.Thread` class and override its `run()` method.

### How it works:

When you extend `Thread`, your class *is* a thread. It carries all the overhead of the Thread class along with your specific task logic.

```java
class MyWorker extends Thread {
    @Override
    public void run() {
        System.out.println("Thread is running: " + Thread.currentThread().getName());
    }
}

public class Main {
    public static void main(String[] args) {
        MyWorker t1 = new MyWorker();
        t1.start(); // This triggers the lifecycle we discussed
    }
}

```

### The "Senior Dev" Verdict:

* **Pros:** Easy to write for small, quick tasks.
* **Cons:** **Inflexible.** Since Java doesn't support multiple inheritance, once you extend `Thread`, you can't extend any other class (like `BaseService` or `Component`).

---

## 2. Implementing the `Runnable` Interface

This is the **preferred approach** in professional software engineering. Instead of defining what the thread *is*, you define what the thread *does*. You separate the **Task** from the **Runner**.

### How it works:

You implement the `Runnable` interface (which has a single method, `run()`) and then pass that instance to a `Thread` object.

```java
class MyTask implements Runnable {
    @Override
    public void run() {
        System.out.println("Task is executing: " + Thread.currentThread().getName());
    }
}

public class Main {
    public static void main(String[] args) {
        MyTask task = new MyTask();
        Thread t1 = new Thread(task); // Pass the task to the thread
        t1.start();
    }
}

```

### The "Senior Dev" Verdict:

* **Pros:** 1.  **Flexibility:** You can still extend another class.
2.  **Decoupling:** Your task logic is separate from thread management.
3.  **Thread Pooling:** `Runnable` is the foundation for modern concurrency tools like `ExecutorService`.
* **Cons:** Slightly more "boilerplate" code (though lambdas fix this).

---

## 3. The Modern Shortcut: Lambdas

Since `Runnable` is a **Functional Interface** (it only has one abstract method), Java 8+ allows us to skip the class declaration entirely using Lambda expressions. This is what you'll see in 90% of modern production codebases.

```java
public class Main {
    public static void main(String[] args) {
        // Lambda style: Concise and readable
        Thread t1 = new Thread(() -> {
            System.out.println("Running in: " + Thread.currentThread().getName());
        });
        
        t1.start();
    }
}

```

---

## 4. Head-to-Head: Thread vs. Runnable

| Feature | Extending `Thread` | Implementing `Runnable` |
| --- | --- | --- |
| **Inheritance** | Uses up your one "extends" slot. | Free to extend another class. |
| **Object Sharing** | Each thread is a unique object. | Multiple threads can share the same `Runnable` instance. |
| **Architecture** | Mixes task and runner logic. | Separates task (Runnable) from runner (Thread). |
| **Suitability** | Simple, isolated scripts. | **Production-grade scalable apps.** |

---

## 5. Critical Reminder: `start()` vs. `run()`

I mentioned this in the lifecycle lecture, but it bears repeating because it's a classic "gotcha" in technical interviews.

* **`thread.start()`**: This tells the JVM to create a **new call stack** and invoke `run()` on that new stack. This is true multithreading.
* **`thread.run()`**: This just calls a normal method. It stays on the **current thread's stack** (usually `main`). No new thread is created.

### Pro-Tip:

If you find yourself creating hundreds of `new Thread()` objects, stop. Creating threads is expensive for the OS. In the next level of this lecture, we would look at **Thread Pools**, where we keep a few workers alive and just feed them `Runnable` tasks. It’s like a kitchen—you don't hire a new chef for every order; you have 5 chefs who handle 100 orders.
